# 태스크 분해 스킬 (Task Breakdown)

화면 분석 결과를 FE 개발 관점에서 실행 가능한 단위 태스크로 분해한다.
사소한 요구사항(빈 상태 UI, 로딩 스켈레톤, 에러 핸들링, 접근성 등)도 모두 태스크로 포함한다.

## 태스크 분해 원칙

1. **독립 실행 가능**: 각 태스크는 다른 태스크와 독립적으로 완료 가능한 단위
2. **예상 시간 2h 이하**: 단일 태스크가 4h 초과 시 더 분해한다
3. **완료 기준 명확**: "완료"를 객관적으로 확인할 수 있어야 한다
4. **레이어 명시**: FSD 레이어(Page/Widget/Feature/Entity/Shared)를 태스크에 표기한다

## 태그 규칙

| 태그 | 대상 |
|------|------|
| `[Page]` | 페이지 컴포넌트, 라우팅 |
| `[Widget]` | 위젯 레이어 컴포넌트 |
| `[Feature]` | 피처 레이어 (액션, 폼, 필터) |
| `[Entity]` | 엔티티 레이어 (데이터 UI) |
| `[Shared]` | 공통 UI 컴포넌트, 유틸 |
| `[API]` | TanStack Query 훅, API 클라이언트 |
| `[State]` | 전역 상태(Zustand), 클라이언트 상태 |
| `[Style]` | 스타일링, 반응형, 애니메이션 |
| `[A11y]` | 접근성 (aria, 키보드 내비게이션) |
| `[Perf]` | 성능 최적화 (메모이제이션, 지연 로딩) |
| `[Test]` | 컴포넌트 테스트, E2E 테스트 |

## 출력 템플릿

```markdown
# [기능명] FE 개발 태스크 목록

## 에픽(Epic): [대분류 기능]

### 스프린트 1 — 기반 구축
- [ ] [Shared] 공통 Button/Input/Modal 컴포넌트 구현 (예상: 2h)
- [ ] [Shared] API 클라이언트 설정 (ky 인스턴스, 인터셉터) (예상: 1h)
- [ ] [API] `use[도메인]Query` 훅 구현 (queryKey, useQuery) (예상: 1h)
- [ ] [API] `use[도메인]Mutation` 훅 구현 (useMutation, 낙관적 업데이트) (예상: 2h)

### 스프린트 2 — 핵심 화면
- [ ] [Page] [화면명] 페이지 컴포넌트 구현 (예상: 1h)
- [ ] [Widget] [위젯명] 위젯 구현 (예상: 2h)
- [ ] [Entity] [엔티티명] 카드/아이템 컴포넌트 구현 (예상: 2h)
- [ ] [Style] 로딩 스켈레톤 UI 구현 (예상: 1h)
- [ ] [Style] 빈 상태(Empty State) UI + CTA 구현 (예상: 1h)
- [ ] [Style] 에러 상태 UI + 재시도 버튼 구현 (예상: 1h)

### 스프린트 3 — 인터랙션 및 폼
- [ ] [Feature] [폼명] 폼 구현 (react-hook-form + zod) (예상: 3h)
- [ ] [Feature] [필터명] 필터 기능 구현 (예상: 2h)
- [ ] [State] [상태명] Zustand 스토어 구현 (예상: 1h)

### 스프린트 4 — 품질 개선
- [ ] [A11y] 키보드 내비게이션 및 aria 속성 추가 (예상: 2h)
- [ ] [Perf] 이미지 지연 로딩, 무한 스크롤 최적화 (예상: 2h)
- [ ] [Test] 핵심 컴포넌트 단위 테스트 작성 (예상: 3h)

## 태스크 의존성 다이어그램

```
[Shared UI] → [Entity] → [Widget] → [Page]
     ↓             ↓
  [API 훅] → [Feature]

## 작성 원칙

- 빈 상태, 로딩 상태, 에러 상태 태스크는 항상 별도로 분리한다
- 반응형 구현이 필요하면 `[Style]` 태그로 별도 태스크화한다
- 접근성 작업은 별도 태스크가 아니라 각 컴포넌트 구현 시 포함하되, 복잡한 경우 `[A11y]` 태스크로 분리한다
- 예상 시간 합계를 스프린트별로 합산해 현실적인 계획인지 검토한다
- 모든 출력은 한국어로 작성한다
