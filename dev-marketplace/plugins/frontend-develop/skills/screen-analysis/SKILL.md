# 화면 분석 스킬 (Screen Analysis)

기획 문서에서 필요한 화면을 도출하고, 각 화면의 컴포넌트 구조와 상태를 설계한다.
사소한 요구사항(로딩 상태, 빈 상태, 에러 상태, 권한 분기 등)도 빠짐없이 포함한다.

## 분석 프로세스

### 1단계: 핵심 사용자 흐름 파악
- 주요 사용자 액션(User Journey)을 3~5개 시나리오로 정리한다
- 각 시나리오에서 사용자가 거치는 화면을 순서대로 나열한다
- 인증/비인증 분기, 권한 분기를 명확히 구분한다

### 2단계: 화면 목록 도출
기획서에서 아래 기준으로 화면을 추출한다:
- 명시적으로 언급된 페이지/화면
- 암묵적으로 필요한 화면 (404, 권한 없음, 로딩, 빈 상태)
- 모달/드로어/시트 (독립 상태로 관리되는 경우)
- 인증 관련 화면 (로그인, 회원가입, 비밀번호 재설정 등)

### 3단계: 컴포넌트 계층 설계
각 화면을 FSD 레이어 기준으로 분해한다:
- **Page**: 라우트와 1:1 매핑되는 최상위 컴포넌트
- **Widget**: 독립적인 UI 블록 (헤더, 사이드바, 카드 섹션 등)
- **Feature**: 사용자 액션 단위 (폼 제출, 필터링, 삭제 등)
- **Entity**: 도메인 데이터 표현 (유저 카드, 상품 아이템 등)
- **Shared**: 범용 UI (버튼, 인풋, 아이콘 등)

### 4단계: 상태 목록 정의
각 화면/컴포넌트의 가능한 상태를 모두 열거한다:
- **로딩 상태**: 데이터 패칭 중 (스켈레톤 UI)
- **성공 상태**: 데이터 정상 표시
- **에러 상태**: API 실패, 네트워크 오류
- **빈 상태 (Empty State)**: 데이터가 없을 때
- **권한 없음**: 접근 불가 상태
- **폼 상태**: idle / 입력 중 / 유효성 오류 / 제출 중 / 완료

## 출력 템플릿

```markdown
# [서비스명] 화면 분석

## 사용자 흐름 요약
1. [흐름 1]: 화면A → 화면B → 화면C
2. [흐름 2]: 화면A → 화면D

---

## 화면 목록

### 📄 [화면명] (`/경로`)
- **목적**: 이 화면의 역할 한 줄 설명
- **진입 조건**: 인증 필요 여부, 권한 조건
- **주요 컴포넌트**:
  - Widget: `[위젯명]` — 역할
  - Feature: `[피처명]` — 역할
  - Entity: `[엔티티명]` — 역할
- **화면 상태**:
  - [ ] 로딩 (스켈레톤)
  - [ ] 성공
  - [ ] 에러 (재시도 버튼)
  - [ ] 빈 상태 (CTA 포함)
- **사용자 액션**: 버튼 클릭, 폼 제출, 필터 변경 등

---

## 전체 화면 목록 요약

| 화면명 | 경로 | 인증 | 우선순위 |
|--------|------|------|---------|
| [화면명] | `/경로` | 필요/불필요 | P0/P1/P2 |

---

## 공통 컴포넌트 (Shared)
- 범용 UI 컴포넌트 목록 (버튼, 인풋, 모달 등)
```

## 작성 원칙

- 기획서에 명시되지 않은 상태(로딩, 에러, 빈 상태)도 반드시 포함한다
- 모바일/데스크톱 분기가 있으면 별도 화면으로 명시한다
- 모달/드로어도 독립 화면으로 취급한다
- 모든 출력은 한국어로 작성한다
- 화면 분석 후 반드시 task-breakdown 스킬을 연계한다
